# 차량 내 어린이/애완동물 감지 및 알림 시스템 요구사항

## 1. 목적

본 시스템은 차량 실내에 어린이 또는 애완동물이 남아 있는 상황을 감지하고, 위험도가 높다고 판단되면 보호자 또는 지정 수신자에게 알림을 제공한다. 초기 구현은 Python 기반 소프트웨어 로직, 센서 입력 추상화, 감지 판단 정책, 알림 정책, 이벤트 기록을 검증 가능하게 구성하는 것을 목표로 한다.

## 2. 범위

### 포함 범위

- 차량 실내 점유 객체 감지 결과 수신
- 어린이 또는 애완동물 존재 가능성 판단
- 차량 상태, 실내 온도, 경과 시간 기반 위험도 산정
- 단계별 알림 발생
- 이벤트 로그 생성
- 테스트 가능한 Python 모듈 구조

### 제외 범위

- 실제 카메라 모델 학습
- 실제 차량 CAN 통신 구현
- 상용 푸시 알림 서비스 연동
- 영상 원본 장기 저장
- 응급기관 자동 신고

## 3. 용어

| 용어 | 설명 |
| --- | --- |
| 점유 대상 | 차량 실내에 남아 있을 수 있는 어린이 또는 애완동물 |
| 감지 입력 | 카메라, 열 센서, 무게 센서, 소리 센서, 차량 상태 등 외부 입력 |
| 위험도 | 점유 대상 존재 가능성, 온도, 차량 잠금 상태, 경과 시간을 종합한 등급 |
| 알림 단계 | 주의, 경고, 긴급으로 구분되는 사용자 알림 수준 |

## 4. 기능 요구사항

| ID | 요구사항 | 수용 기준 | 우선순위 |
| --- | --- | --- | --- |
| FR-001 | 시스템은 센서별 감지 입력을 공통 데이터 모델로 수신해야 한다. | 카메라, 온도, 차량 상태 입력을 동일한 이벤트 처리 파이프라인에 전달할 수 있다. | Must |
| FR-002 | 시스템은 어린이 존재 가능성을 판단해야 한다. | 입력 이벤트에 `child_probability`가 있을 때 정책 임계값에 따라 어린이 감지 상태를 반환한다. | Must |
| FR-003 | 시스템은 애완동물 존재 가능성을 판단해야 한다. | 입력 이벤트에 `pet_probability`가 있을 때 정책 임계값에 따라 애완동물 감지 상태를 반환한다. | Must |
| FR-004 | 시스템은 차량 잠금 상태를 위험도 산정에 반영해야 한다. | 차량이 잠긴 상태이고 점유 대상이 감지되면 위험도가 상승한다. | Must |
| FR-005 | 시스템은 실내 온도를 위험도 산정에 반영해야 한다. | 설정된 고온 또는 저온 임계값을 벗어나면 위험도가 상승한다. | Must |
| FR-006 | 시스템은 점유 대상 감지 후 경과 시간을 위험도 산정에 반영해야 한다. | 설정된 지속 시간 이상 감지되면 알림 단계가 상향된다. | Must |
| FR-007 | 시스템은 위험도에 따라 주의, 경고, 긴급 알림을 생성해야 한다. | 동일 입력 조건에서 결정적 알림 단계를 반환한다. | Must |
| FR-008 | 시스템은 중복 알림을 억제해야 한다. | 동일 위험 이벤트에 대해 설정된 쿨다운 시간 내 반복 알림을 보내지 않는다. | Must |
| FR-009 | 시스템은 알림 수신자 목록을 설정으로 관리해야 한다. | 수신자 추가, 제거, 조회가 가능하고 빈 수신자 목록은 검증 오류를 낸다. | Should |
| FR-010 | 시스템은 감지 이벤트와 알림 이벤트를 감사 로그로 남겨야 한다. | 이벤트 ID, 시각, 위험도, 알림 단계, 처리 결과가 기록된다. | Must |
| FR-011 | 시스템은 외부 알림 채널을 어댑터로 분리해야 한다. | 테스트에서는 네트워크 호출 없이 가짜 알림 어댑터로 검증할 수 있다. | Must |
| FR-012 | 시스템은 감지 임계값을 설정 파일 또는 환경 변수로 변경할 수 있어야 한다. | 코드 수정 없이 임계값 변경 후 테스트에서 다른 판단 결과를 확인할 수 있다. | Should |
| FR-013 | 시스템은 센서 입력 누락 또는 형식 오류를 명시적으로 처리해야 한다. | 잘못된 입력은 예외 또는 실패 결과로 반환되고 감사 로그에 남는다. | Must |
| FR-014 | 시스템은 위험 이벤트의 현재 상태를 조회할 수 있어야 한다. | 마지막 감지 상태, 위험도, 마지막 알림 시각을 반환한다. | Should |
| FR-015 | 시스템은 시뮬레이션 입력으로 전체 정책 흐름을 검증할 수 있어야 한다. | 통합 테스트에서 감지 입력부터 알림 생성까지 재현 가능하다. | Must |

## 5. 비기능 요구사항

| ID | 요구사항 | 수용 기준 | 우선순위 |
| --- | --- | --- | --- |
| NFR-001 | 판단 로직은 결정적이어야 한다. | 동일 입력과 동일 설정은 항상 동일한 위험도와 알림 결과를 반환한다. | Must |
| NFR-002 | 핵심 정책 판단은 빠르게 수행되어야 한다. | 단일 이벤트 처리 시간은 일반 개발 장비 기준 100ms 이하를 목표로 한다. | Should |
| NFR-003 | 개인정보 보호를 기본값으로 적용해야 한다. | 원본 영상, 음성, 식별 가능한 개인정보는 저장하지 않는다. | Must |
| NFR-004 | 로그는 민감 정보를 최소화해야 한다. | 로그에는 원본 이미지, 위치 세부값, 연락처 원문을 남기지 않는다. | Must |
| NFR-005 | 시스템은 외부 서비스 장애에 견고해야 한다. | 알림 어댑터 실패 시 재시도 가능 상태를 기록하고 전체 프로세스가 비정상 종료되지 않는다. | Must |
| NFR-006 | 시스템은 테스트 가능해야 한다. | 외부 입출력은 인터페이스로 분리되어 단위 테스트에서 대체 가능하다. | Must |
| NFR-007 | 시스템은 설정 오류를 시작 시점에 검출해야 한다. | 필수 설정 누락 또는 잘못된 임계값은 명시적 오류로 보고된다. | Should |
| NFR-008 | 코드 품질 검증을 자동화해야 한다. | PR마다 테스트, 커버리지, lint, format, type check가 실행된다. | Must |
| NFR-009 | 시스템은 Python 3.11 이상에서 동작해야 한다. | CI가 Python 3.11 및 3.12에서 검증된다. | Must |
| NFR-010 | 알림 정책 변경은 추적 가능해야 한다. | 정책 변경 PR에는 관련 요구사항 ID와 테스트 ID가 포함된다. | Should |
| NFR-011 | 시스템은 장애 원인 분석이 가능해야 한다. | 주요 결정 지점마다 구조화된 이벤트 로그를 남긴다. | Should |
| NFR-012 | 안전 관련 제한사항을 명시해야 한다. | 릴리즈 노트에 알려진 오탐, 미탐, 환경 제한을 기록한다. | Must |

## 6. TDD 기준

| 요구사항 유형 | 테스트 기준 |
| --- | --- |
| 감지 판단 | 단위 테스트로 임계값 경계 조건을 검증한다. |
| 위험도 산정 | 차량 잠금, 온도, 경과 시간 조합 테스트를 작성한다. |
| 알림 정책 | 중복 억제, 단계 상승, 실패 처리 테스트를 작성한다. |
| 설정 | 정상 설정과 오류 설정을 모두 테스트한다. |
| 통합 흐름 | 시뮬레이션 입력에서 알림 이벤트까지 검증한다. |

- 모든 Must 요구사항에 대응하는 테스트가 존재한다.
- PR 검증 워크플로가 통과한다.
- 안전 관련 알려진 제한사항이 `RELEASE.md`에 기록된다.
- 민감 정보 저장 여부를 점검한다.
